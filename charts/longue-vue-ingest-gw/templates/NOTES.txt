longue-vue-ingest-gw {{ .Chart.AppVersion }} installed as release {{ .Release.Name }} in namespace {{ .Release.Namespace }}.

This chart deploys the DMZ ingest gateway for the longue-vue CMDB (ADR-0016). It
fronts longue-vue's mTLS-only ingest listener and forwards write requests from
the K8s push collector across the trusted-zone boundary.

Configuration summary
─────────────────────

  Inbound listener:    :{{ .Values.listener.port }}    (TLS, fronted by Envoy in production)
  Health/metrics:      :{{ .Values.listener.health.port }}    (plaintext, scraped by Prometheus)
  Upstream:                {{ .Values.upstream.url }}
  mTLS source mode:    {{ .Values.mtls.mode }}
  Verify cache:        {{ .Values.cache.ttl }} (positive) / {{ .Values.cache.negativeTtl }} (negative), max {{ .Values.cache.maxEntries }} entries
  Required scope:      {{ .Values.requiredScope | default "none" }}

Quick checks
────────────

  1. Pods are Running and Ready:

       kubectl -n {{ .Release.Namespace }} get pods -l app.kubernetes.io/instance={{ .Release.Name }}

  2. Cert is loaded and not close to expiry. Replace POD with one of the
     pods from the previous command:

       kubectl -n {{ .Release.Namespace }} exec POD -c ingest-gw -- \
         wget -qO- http://localhost:{{ .Values.listener.health.port }}/metrics \
         | grep -E '^longue_vue_ingest_gw_cert_(not_after|reload)'

  3. Upstream longue-vue reachable (readyz returns 200 only when the gateway
     can complete an mTLS handshake against longue-vue's /healthz):

       kubectl -n {{ .Release.Namespace }} port-forward svc/{{ include "longue-vue-ingest-gw.fullname" . }} {{ .Values.listener.port }}:{{ .Values.listener.port }}
       curl -k https://localhost:{{ .Values.listener.port }}/readyz

{{- if eq .Values.mtls.mode "vault" }}

Vault PKI mode notes
────────────────────

  • The Vault Agent injector mutates the pod spec at admission time. If
    pods stay in PodInitializing after install, check that:
    - the injector is installed in the cluster
    - the Vault role "{{ .Values.mtls.vault.role }}" exists and is bound
      to ServiceAccount "{{ include "longue-vue-ingest-gw.serviceAccountName" . }}"
      in namespace "{{ .Release.Namespace }}"
    - the Vault PKI role "{{ .Values.mtls.vault.pkiRole }}" can issue certs
      with TTL {{ .Values.mtls.vault.certTTL }}

  • Cert TTL is {{ .Values.mtls.vault.certTTL }}. Vault Agent re-fetches at
    {{ .Values.mtls.vault.renewAt }}% of TTL. fsnotify in the gateway
    re-reads on file change — no pod restart needed.
{{- else if eq .Values.mtls.mode "secret" }}

Secret mode notes
─────────────────

  • mTLS client cert is read from Secret "{{ .Values.mtls.secret.name }}"
    in namespace "{{ .Release.Namespace }}". The Secret MUST be of type
    kubernetes.io/tls (cert under tls.crt, key under tls.key).

  • cert-manager users: point a Certificate resource at this Secret name
    and the gateway will reload automatically on every renewal.

  • Manual rotation: write the new cert/key into the Secret, no pod
    restart required — fsnotify in the gateway re-reads on file change.
{{- else if eq .Values.mtls.mode "file" }}

File mode notes
───────────────

  • mTLS client cert is mounted from the volume specified in
    .Values.mtls.file.volume — operator is responsible for keeping the
    cert/key files at /etc/longue-vue-ingest-gw/tls/{tls.crt,tls.key} fresh.

  • This mode is the escape hatch for environments without Vault and
    without standard k8s Secret tooling. If you can use mode=secret,
    prefer it.
{{- end }}
